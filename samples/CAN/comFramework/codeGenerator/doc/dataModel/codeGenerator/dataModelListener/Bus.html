codeGenerator.dataModelListener

Class Bus

  • java.lang.Object
    • codeGenerator.dataModelListener.NetObject
      • codeGenerator.dataModelListener.Bus

  • public class Bus
extends NetObject
    The data structure describing a communication bus in the cluster. An object of this class contains the results of parsing a single DBC input file. The object is part of the nested data structure passed to the template engine for rendering the information in the wanted format.

    The basic design principle is the use of public members because such members can be directly accessed from the output templates. Boolean decisions are either expressed as explicit public Boolean members or by references to nested member objects, which are null if an optional member is not used or applicable: StringTemplate can query public Boolean members or check for null references.

      • networkFile

        public FileExt networkFile
        The name of the network description file, which has been parsed into this object. The network file is described with an extended Java File object. This object extends the Java class File by adding some information about parts of the file name (like its extension). Please refer to class FileExt.

        The information from the Java class File is partly accessible: All methods starting with get or is can be used by placing the method name without the syllable get/is into the template, e.g. would yield the return value of File.getName().

        Please refer to the online help of Java class File for details.

      • sortFrameAry

        public final codeGenerator.dataModelListener.ST4CmdInterpreter<java.lang.Object,java.lang.Object> sortFrameAry
        This field can be used to demand a certain sort order of the frame list frameAry. The order of frames is changed by passing the wanted sort order in a StringTemplate V4 template expression like <bus.sortFrameAry.byId>, where byId is meant an example of a supported sort order. All subsequently evaluated template expressions will find the field <bus.frameAry> in the new sort order. Supported sort orders are:

        • byName: Ascending lexical order of names
        • byNameInverse: Descending lexical order of names
        • byId: Ascending CAN ID
        • byIdInverse: Descending CAN ID
        • bySize: Ascending size of payload
        • bySizeInverse: Descending size of payload
        • sentFirst: Sent frames before received frames
        • receivedFirst: Received frames before sent frames

        Some sort orders, e.g. sorting by size of payload, are undecided for several frames. A secondary, sub-ordinated sort criterion can be implemented by sorting twice. Begin with the sub-ordinated criterion (e.g. sorting by CAN ID) followed by the principal sort order (e.g. by size of payload). In this example would larger frames follow smaller ones and all equally sized would be ordered by ascending CAN ID.

        Please note, once you've sorted the array of frames the initial order is lost and can't be restored again. This is the order of appearance in the network file. However, a frame iteration in this order is still possible via field Cluster.frameInFileOrderAry. Reordering a bus' frame list won't affect the frame lists owned by Cluster; Cluster.frameByNameAry, Cluster.frameByIdAry and Cluster.frameInFileOrderAry are constant throughout the complete template expansion process.

      • isTransmissionDirectionInverted

        public boolean isTransmissionDirectionInverted
        The transmission direction of frames (sent, received, none of these or both) is encoded in the network database and can be found in according fields in Frame, Pdu and Signal. However, on command line demand can this information be inverted in order to support code generation for residual bus simulation. Actually sent frames are reported as being received and vice versa. (See command line argument invert-transmission-direction.)

        This field reflects the value of the command line argument. A StringTemplate V4 template can be made aware of transmission direction inversion.

        Remark:
        Caution, the inversion of the transmission direction only relates to the fields isSent and isReceived of frames, PDUs and signals. The textual information about senders and receivers is not affected. In this sense is the resulting data model inconsistent if inversion is applied.

      • hasFloatingPointSignals

        public boolean hasFloatingPointSignals
        Theoretically a signal may have a floating point data type. Since this is very rare in practice any normal application of the code generator will likely not support this case. To enable safe code generation templates here's a flag which tells if there is at least one floating point signal in use in the bus. If so, the template could generate an error.

      • hasScaledFloatingPointSignals

        public boolean hasScaledFloatingPointSignals
        Theoretically a floating point signal in a network database file can have a linear scaling. However, this is completely useless and won't probably be ever supported by any code generation template. To enable safe code generation templates here's a flag which tells if there is at least one scaled floating point signal in use in the bus. If so, the template could generate an error.

      • optionMap

        public java.util.HashMap<java.lang.String,java.lang.Object> optionMap
        A map of user specified options or template attributes. User options are template attributes, which are simply passed through from the application's command line to the StringTemplate V4 template and can there be used to control the code generation. The use case are optional constructs in the generated code, which are controlled from the command line.

        User options/attributes can be given in the global context of the command line or in the context of a bus or an output file. This map holds all options given in the context of the bus. (Refer to Info.optionMap for the output file related options.)

        User options, which are set in the global context of the command line are adopted by all buses. Nonetheless, a particular bus can re-specify the same option with deviating value.

        The name of an option or template attribute is the key into the map and the attribute's value is the value of the map entry. The value object's Java type is one out of String, Boolean, Integer or Double, depending on which fits best to the original command line argument; a command line argument cmdLineArg with value true would obviously be passed as Boolean into the template and would permit conditional code by using a construct like: <if(bus.optionMap.cmdLineArg)>Command line argument is TRUE!<endif>
