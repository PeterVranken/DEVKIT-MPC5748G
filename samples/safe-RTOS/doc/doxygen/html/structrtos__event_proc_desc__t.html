Detailed Description

The representation of a task triggering event processor.
Note, to support APIs for configuration at runtime, the object contains both, runtime information and static configuration data. A resource optimized implementation could separate both parts and it could then put the configuration data into ROM. (Instead of offering the run-time configuration by APIs.) This holds for the task representation, too.

◆ minPIDForTrigger

uint8_t rtos_eventProcDesc_t::minPIDForTrigger

An event can be triggered by user code, using rtos_sendEvent(). However, tasks belonging to less privileged processes must not generally granted permission to trigger events that may activate tasks of higher privileged processes. Since an event is not process related, we make the minimum process ID explicitly configured, which is required to trigger this event.
Only tasks belonging to a process with PID >= minPIDToTriggerThisEvProc are permitted to trigger this event.
The range of minPIDToTriggerThisEvProc is 0 ... (RTOS_NO_PROCESSES+1). 0 and 1 both mean, all processes may trigger the event, RTOS_NO_PROCESSES+1 means only OS code can trigger the event.

◆ tiCycleInMs

unsigned int rtos_eventProcDesc_t::tiCycleInMs

The period time of the (cyclic) event in ms. The permitted range is 0..2^30-1.
0 means no regular, timer controlled activation. The event is only enabled for software trigger using rtos_sendEvent() (by interrupts or other tasks).

◆ priority

unsigned int rtos_eventProcDesc_t::priority

The priority of the event (and thus of all associated user tasks, which inherit the prioritry) in the range 1..UINT_MAX. Different events can share the same priority. If they do then the execution of their associated tasks will be sequenced when they become due at same time or with overlap.
Note, if the event has a priority above RTOS_MAX_LOCKABLE_TASK_PRIORITY then only those tasks can be associated, which belong to the process with highest PID in use. This is a safety constraint.

◆ noActivationLoss

unsigned int rtos_eventProcDesc_t::noActivationLoss

We don't queue event triggers. If at least one task is still busy, which had been activated by the event at its previous due time then an event sent with rtos_sendEvent() (and the activation of the associated tasks) can be lost:
If using arbitrary task parameters, then the event and the next task activation is lost.
If using countable events then the event sent with rtos_sendEvent() is counted and stored and will lead to a task activation later. All counted and stored events will be handed over to the associated tasks at once. No task activation loss is recorded. However, if the counter for events saturates at its implementation maximum then no new event can be accounted and a task activation loss is recorded.
This is counter of lost task activations, intended for diagnostic purpose. The counter is not saturated, it won't wrap around at its implementation maximum; this behavior allows most easy evaluation of the rate of events.

Remarks
This field is shared with the external client code of the module. There, it is read only. Only the scheduler code must update the field.
